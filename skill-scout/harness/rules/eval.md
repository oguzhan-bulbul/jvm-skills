# Skill-Scout evaluation rules

Plain-English guidance the judge + bundle-eval agents read **before** judging. The rule-refinement
agent appends concise, general learnings here after each conference (newest changelog entry on top).
Keep rules general and non-redundant. Git-tracked — every change is auditable and revertable.

## What we are looking for

A directory of **created, reusable, opinionated skills** authored by the speaker in their OWN
(non-fork) repo. Prefer recall on genuinely good skills; reject noise with a specific reason.

- **found** — a real skill with concrete, opinionated guidance (Depth >= 2, ~100+ lines), authored
  by this person, not a demo/talk fixture. JVM skills are the primary target.
- **needs_review** — real but borderline (thin, demo-ish, niche, blog-coupled). When genuinely
  unsure whether something real is promotable, prefer needs_review over reject — never silently drop.
- **reject** — everything else, with a reason from the taxonomy.

## Core principles (seeded from review)

1. **Demo/workshop repo is NOT a reason to reject.** Judge the skill *content's* reusability, not the
   repo's purpose. Several strong authors (e.g. Anton Arhipov) ship genuinely reusable skills *via*
   demo/workshop/conference repos. A 200-line Spring Batch skill in `sdd-demo` is still a real skill.
2. **A copied skill is promoted ONCE, at its canonical repo.** When the same skill appears in several
   of an author's repos, the canonical = the dedicated collection repo, else most stars, else non-demo.
   Don't let the same skill land in both the found and rejected piles.
3. **Bare `CLAUDE.md` / `AGENTS.md` / `.cursorrules` = `project-doc`**, even when the topic is JVM.
   If the file is only build/run/test commands, project layout, or workshop flow — not transferable
   skill guidance — it is `project-doc`, not a skill. (e.g. a Kafka *training project's* CLAUDE.md.)
4. **Numbered SKILL.md in one skills dir = a BUNDLE.** `01-spec`, `02-criteria`, … `06-execute` are
   steps of one pipeline and are meaningless alone. Evaluate the suite as a unit (see bundles below);
   never reject the steps individually as "generic workflow."
5. **Workflow skills can be listed.** A general agentic/dev workflow (code review, spec-driven dev,
   PR defense) is not JVM, but a *coherent, high-quality* workflow bundle is directory-worthy under
   the `workflow` category. Evaluate these MORE extensively — read the members, judge coherence and
   reusability — before deciding. A single thin workflow step on its own is still `off-topic-workflow`.

## Strong JVM signals (high jvm_fit)

Spring (Batch / Boot / Data / Framework), JFR / Java Flight Recorder, Testcontainers, JPA / Hibernate,
jOOQ, Quarkus, Micronaut, Ktor, Kotlin language features, Gradle / Maven internals, GraalVM, JBang,
Jakarta EE, JUnit, Reactor / Project Reactor, Vert.x, Android / Jetpack Compose / Compose Multiplatform /
Kotlin Multiplatform (KMP). Kafka counts as JVM only when the guidance is about JVM clients/Streams, not
generic ops.

## Not JVM (reject reasons)

- `off-topic-workflow` — real reusable skill, generic agent/dev workflow, not JVM (review-orchestration,
  spec-driven-dev step, comms style, thread handoff).
- `off-topic-service` — real skill for an external service (Stripe, HubSpot, Zoom, Google, Tavily).
- `off-topic-tech` — real skill for non-JVM tech (CSS/Tailwind/daisyUI, React/Vue, Python/Rust/Go, robotics).
- `demo` — example/workshop/sample/newsletter skill with no reusable guidance.
- `vendored` — third-party skill copied into the repo, not authored by the speaker (chase the real author).
- `test-fixture` — a skill under `src/test/...`, a fixture not an authored skill.
- `boilerplate` — duplicated template family (OpenSpec, Tessl iikit).
- `jvm-collection` — a real JVM skill that is one member of a found collection (promote-worthy sibling).

## Reasoning requirement

Every verdict needs specific, substantive reasoning (1–2 sentences): name what the skill actually
does/teaches, the author + repo signal (stars, fork), and why it lands in this bucket. Never a generic
template like "demo repo" or "real SKILL.md".

- 2026-07-03 (JCConf Taiwan 2025) — **A verdict that contradicts its own reasoning text must not be trusted at face value — treat the judge's hedge words as a stronger signal than its own status/score.** When the judge's own `reasoning`/`notes` field contains phrases like "language-agnostic", "context-free", "transferable methodology but [X]-focused, not JVM", or names non-JVM frameworks as equally applicable, that is the judge self-flagging the JVM-fit failure — override to reject even when it still emitted `status=found` (sometimes alongside `jvm_fit=high`, sometimes even alongside its own `jvm_fit=low`). - 2026-06-28 (Java Forum Nord 2025) — **Agent-company orchestration state references are a framework-coupling signal that fails the extractability test, even when the skill's JVM topic (Micronaut, GraalVM) is genuine.** When a SKILL.md's "Session Start", "First Read", or procedure sections assume a running multi-agent company runtime — referencing constructs like `executionState.returnAssignee`, `stage participants`, `active-issue/execution-stage/AGENTS.md`, or internal agent-routing logic — the skill cannot be used outside that pipeline. Classify `project-doc` regardless of JVM depth or line count. This extends the jChampions 2026 rule (numbered-file/script coupling) to agent-framework state-variable coupling: proprietary phase labels and scripts were already covered; orchestration state variables are the new form. (e.g., alvarosanchez/micronaut-agent-company: `skills/micronaut-security-review/SKILL.md` and `skills/micronaut-test-resources-provider-development/SKILL.md` were both promoted by the judge on substantive JVM content; Opus correctly rejected both because their Session Start sections depended on Paperclip pipeline state — active DEV issues, execution stages, returnAssignee routing — that exists only inside the agent company runtime, not as standalone guidance.)
- 2026-06-28 (W-JAX 2025) — **A SKILL.md that delegates all technical substance to external companion files (e.g., `references/NNN-topic.md`) is a dispatch shell — not independently promotable.** When the file's own content is only a topic overview plus a workflow step like "Step 1: Read references/NNN.md for patterns/examples," with no embedded JVM expertise, the extractability test fails identically to the jChampions 2026 pipeline-coupling case: the skill cannot be dropped into another project without transplanting the companion reference docs. Do not substitute repository reputation (stars, prominent author, named collection) for absent file content — evaluate what is actually in the file. Classify `project-doc`. (e.g., jabrena/cursor-rules-java generated 51 judge↔recheck disagreements: the judge promoted skills citing "410 stars, professional JVM collection" while Opus correctly rejected all 51 because individual SKILL.md files were thin dispatch wrappers with no embedded JVM guidance, deferring all technical content to unreachable `references/NNN-topic.md` companion files.)
- 2026-06-28 (Jfokus 2026) — **Code-generation skills must have their template import packages verified against the runtime namespace of the declared dependency.** When a skill's core deliverable is compilable Java/Kotlin code templates (pom.xml stubs, Gradle build files, src/ class files), spot-check that the `import` package prefixes in those templates match the runtime namespace published by the declared Maven/Gradle artifact — not just the artifact's existence. A package rename between SDK versions produces a namespace mismatch (e.g., templates `import io.mcp.server.*` while `pom.xml` declares `io.modelcontextprotocol.sdk:mcp` whose actual runtime package is `io.modelcontextprotocol.*`) that makes every generated class fail to compile, rendering the skill's core value non-functional. Classify `hallucinated-api`. This is a structural extension of the existing hallucinated-api rule (which targets method/class names) to cover package-level namespace errors in template files. (e.g., anishi1222/multi-agent-code-reviewer/.github/skills/java-mcp-server-generator/SKILL.md: 756 lines of Java MCP server generation templates — judge promoted at depth 3; Opus recheck correctly rejected because template imports used the old pre-rename package `io.mcp.server.*` instead of the SDK's actual published namespace.)

It does not: the qualifier describes the minimum that still qualifies as project-doc; files with richer content escalate to the Devoxx Poland 2026 extractability test, but the default presumption remains project-doc. - 2026-06-28 (J-Spring 2026) — **A personal philosophy or guiding-principles file housed in a `skills/` directory is not a reusable skill.** Files framed as "zen-of-X", "X's guiding principles", or coding manifestos express an author's beliefs (FP principles, immutability, communication style) rather than teaching a repeatable technique with concrete steps and examples. The reasoning requirement asks "what does/teaches" — a manifesto teaches no actionable method. Classify as `project-doc` (author-internal) or `off-topic-workflow` (generic philosophy), never `found`. Test: does the content prescribe a specific, reproducible *method* with concrete steps or code, or does it declare what the author believes is good? If the latter, reject. Reject the copy as `jvm-collection` (same author) or `vendored` (different author); evaluate the skill at the `import_url` source only. Classify `demo` (if one-off/workshop artifact) or `needs_review` at most; the deprecation signal overrides depth and JVM-fit scores. Distinct from the Digital Crafts Day timestamped-run-archive rule (`experiments/runs/<timestamp>/`): here the signal is the author's own explicit `archive/` folder, not an experiment snapshot path. - 2026-06-28 (Voxxed Days Bucharest 2026) — **Hardcoded OS-specific absolute paths are a non-reusability signal that overrides JVM content quality.** When a skill's "Useful Paths", "Commands", or similar sections embed machine-local absolute paths (e.g., `C:\VoxxedDays\...`, `C:\Users\<name>\...`) tied to one person's local setup, the skill cannot be used by anyone else without modification — classify `demo` directly, regardless of JVM depth, line count, or content quality. - 2026-06-28 (Devoxx Poland 2026) — **Once ≥3 identical files are confirmed across directories of the same repo, issue a single `boilerplate` verdict for all — do not re-evaluate each file individually.** The jPrime 2026 rule identifies the trigger condition, but the judge keeps re-judging each copy independently and repeatedly scoring the same template as "transferable guidance." - 2026-06-28 (JavaLand 2026) — **A skill designed as explicitly polyglot (JVM + another language as co-equal parallel tracks) is not JVM-specific, even if Java is fully covered.** When every workflow step and code example systematically offers "Java/jqwik OR TypeScript/fast-check" (or similar) with equal weight and documentation, JVM content is not "the subject" — the skill is a general polyglot guide with a Java lane.
